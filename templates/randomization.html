How randomization works

Pool

A pool is a set of survey pages from which the system selects one or more to show each participant. You can have multiple independent pools in the same survey — each runs its own balanced assignment.

Show N page(s)

How many pages from the pool each participant sees. With 4 pages and Show = 2, there are 6 possible combinations. Each participant is assigned one combination.

Balanced allocation (minimization)

The algorithm always assigns the combination with the lowest count so far. Ties are broken randomly. This keeps exposure counts as balanced as possible without pure randomization — no participant-level blocking needed.

Condition variable

Only available when Show = 1. Set a name here (e.g. condition) and the system will expose that variable in the survey before it renders — so you can use it in visibleIf expressions on any page:

"visibleIf": "{condition} = 'A' or {condition} = 'C'"

The value is taken from the Value map if provided, or is the page name itself if not.

Value map

Optional JSON that translates page names into readable condition values. Without it, the raw page name becomes the value. Example:

{"info_a": "A", "info_b": "B", "info_c": "C"}

Or with meaningful names:

{"info_a": "control", "info_b": "treatment"}

Multiple pools

Each pool is independent. A survey can have a pool for condition assignment (Show = 1, with condition variable) and a separate pool for vignette order (Show = 2, no condition variable) — they run simultaneously and don't interfere.

Saving a pool resets its counters

Any configuration change resets the assignment counts for that pool. Use Reset to clear counts without changing the configuration.
